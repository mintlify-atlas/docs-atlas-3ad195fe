---
title: "Two-step verification"
description: "How 2SV works in Signon, including mandatory enforcement, admin resets, and exemptions."
---

Two-step verification (2SV) adds a second authentication factor to user sign-in. After entering their password, users are prompted for a time-based one-time password (TOTP) generated by an authenticator app.

## How 2SV works

Signon uses TOTP (RFC 6238). Users scan a QR code with an authenticator app (such as Google Authenticator, Aegis, or any standards-compliant TOTP app) during setup. On subsequent sign-ins, the app generates a 6-digit code that refreshes every 30 seconds.

The OTP secret key is stored encrypted in the database.

## 2SV statuses

| Status | Meaning |
| --- | --- |
| `enabled` | The user has set up 2SV and has an OTP secret key. |
| `not_set_up` | The user has not set up 2SV and does not have an exemption. |
| `exempted` | The user has been granted an exemption and will not be prompted for 2SV. |

You can filter the Users list by 2SV status to identify users who have not yet completed setup.

## Setting up 2SV as a user

<Steps>
  <Step title="Sign in with your password">
    Complete the normal sign-in flow. If 2SV is required but not yet set up, you will be redirected to the setup page automatically.
  </Step>
  <Step title="Scan the QR code">
    Open your authenticator app and scan the QR code displayed on screen, or enter the secret key manually.
  </Step>
  <Step title="Verify the code">
    Enter the 6-digit code shown in the authenticator app to confirm the setup was successful. Signon records the OTP secret and 2SV is now active on the account.
  </Step>
</Steps>

## Mandatory 2SV

### Admin roles

In the **production** environment, Signon automatically sets `require_2sv = true` for any user whose role requires it. The roles that always require 2SV are:

- Superadmin
- Admin
- Super organisation admin
- Organisation admin

This is enforced by a `before_save` callback: when a user's role changes to one of these, `require_2sv` is set to `true` automatically.

<Note>
This automatic enforcement only applies in the production environment. On other environments, 2SV can be configured manually.
</Note>

### Per-organisation enforcement

Organisations can have mandatory 2SV enabled. When an organisation requires 2SV, all users belonging to that organisation must have `require_2sv = true`. Signon validates this at account creation time and will reject a new user record if the organisation requires 2SV but the flag is not set.

Users in a mandatory-2SV organisation who have an exemption are not prompted for 2SV, but the exemption must be explicitly granted (see [2SV exemptions](#2sv-exemptions) below).

## Trusted devices (30-day remember)

After a successful 2SV verification, users can opt to be remembered on that device for **30 days**. During this period, the second factor is not requested again on the same device.

## Resetting a user's 2SV

Admins can reset a user's 2SV, for example when a user has lost access to their authenticator app.

<Steps>
  <Step title="Find the user">
    Navigate to **Users** and search for the user by name or email.
  </Step>
  <Step title="Open the edit form">
    Click on the user, then click **Edit**.
  </Step>
  <Step title="Reset 2SV">
    Click **Reset 2-step verification**. This clears the user's OTP secret key and sets `require_2sv = true`, forcing them to set up 2SV again on their next sign-in.
  </Step>
</Steps>

The reset is recorded in the user's event log, including which admin performed it.

<Warning>
Resetting 2SV does not lock or suspend the account. The user can still sign in with their password immediately, but will be prompted to set up 2SV before they can access applications.
</Warning>

## 2SV exemptions

A 2SV exemption allows a specific user to bypass the 2SV requirement. Exemptions must include a reason and an expiry date.

### Granting an exemption

<Steps>
  <Step title="Open the user's edit form">
    Navigate to **Users**, find the user, and click **Edit**.
  </Step>
  <Step title="Complete the exemption fields">
    Enter a **reason for exemption** and an **expiry date**. Both fields are required; Signon will reject the form if either is missing.
  </Step>
  <Step title="Save the exemption">
    Click **Update user**. The exemption is applied immediately and recorded in the event log.
  </Step>
</Steps>

When an exemption is granted:
- `require_2sv` is set to `false`
- The user's existing OTP secret key is cleared
- `reason_for_2sv_exemption` and `expiry_date_for_2sv_exemption` are recorded

<Note>
Admin roles (superadmin, admin, super organisation admin, organisation admin) cannot be exempted from 2SV. Attempting to save an exemption for a user with one of these roles will produce a validation error.
</Note>

### Updating or removing an exemption

To update an existing exemption, edit the reason or expiry date and save. The event log records the change.

To remove an exemption, clear the reason field. If `require_2sv` is set to `true` for the user (for example due to their role or organisation), the exemption fields are automatically cleared on save.

## Account locking after failed 2SV attempts

After **10 consecutive failed** 2SV code attempts, the account is automatically locked. The lock reason is recorded as `:two_step` to distinguish it from a password-based lock.

A locked account must be unlocked by an admin before the user can sign in again. See [Suspensions and unlocking](/user-management/suspensions-and-unlocking) for instructions.
