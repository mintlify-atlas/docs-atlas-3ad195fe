---
title: "Authorization Code flow"
description: "Step-by-step guide to the Authorization Code grant type used for end-user authentication in Signon."
---

When an end-user accesses a GOV.UK admin application without an active session, the application redirects them through Signon to authenticate. This uses the **Authorization Code** grant type (OAuth2 section 5.1).

Only `authorization_code` is enabled in Signon's Doorkeeper configuration. The implicit, password, and client credentials grant types are not available.

## Flow walkthrough

<Steps>
  <Step title="User requests a protected resource">
    The user visits a protected page in a client application. The application detects there is no valid session and begins the OAuth2 flow.
  </Step>
  <Step title="Application redirects to Signon">
    The application sends the user's browser to the Signon authorization endpoint. The `state` parameter is required.

    ```http
    GET /oauth/authorize?response_type=code
      &client_id=s6BhdRkqt3
      &redirect_uri=https%3A%2F%2Fmyapp.com%2Fauth%2Fgds%2Fcallback
      &state=xyz HTTP/1.1
    Host: signon.publishing.service.gov.uk
    ```

    <Note>
      The `state` value must be an unguessable random string generated by the client. Signon rejects authorization requests that omit it.
    </Note>
  </Step>
  <Step title="Signon authenticates the user">
    Signon checks whether the user has an active session. If not, it presents the sign-in page (and two-step verification if required).

    Once authenticated, Signon checks that the user has the `signin` permission for the requesting application. If they do not, they are redirected to a "sign-in required" page rather than being granted a code.
  </Step>
  <Step title="Signon redirects back with an authorization code">
    On success, Signon redirects the user's browser to the client's `redirect_uri` with a short-lived authorization code and the original `state` value.

    ```http
    HTTP/1.1 302 Found
    Location: https://myapp.com/auth/gds/callback
      ?code=SplxlOBeZQQYbYS6WxSbIA
      &state=xyz
    ```

    The client must verify that the returned `state` value matches what it sent.
  </Step>
  <Step title="Application exchanges the code for an access token">
    The client makes a back-channel POST request to exchange the authorization code for an access token. Client credentials are sent in the request body.

    ```http
    POST /oauth/token HTTP/1.1
    Host: signon.publishing.service.gov.uk
    Content-Type: application/x-www-form-urlencoded;charset=UTF-8

    grant_type=authorization_code
      &code=SplxlOBeZQQYbYS6WxSbIA
      &client_id=s6BhdRkqt3
      &client_secret=7Fjfp0ZBr1KtDRbnfVdmIw
      &redirect_uri=https%3A%2F%2Fmyapp.com%2Fauth%2Fgds%2Fcallback
    ```

    Signon responds with the access token:

    ```json
    {
      "access_token": "de6780bc506a0446309bd9362820ba8aed28aa506c71eedbe1c5c4f9dd350e54",
      "token_type": "bearer",
      "expires_in": 7200,
      "refresh_token": "8257e65c97202ed1726cf9571600918f3bffb2544b26e00a8a5d5cfe5a04d8",
      "created_at": 1680000000
    }
    ```
  </Step>
  <Step title="Application fetches user details">
    The client uses the access token to retrieve the authenticated user's details and permissions from the `/user` endpoint.

    ```http
    GET /user HTTP/1.1
    Host: signon.publishing.service.gov.uk
    Authorization: Bearer de6780bc506a0446309bd9362820ba8aed28aa506c71eedbe1c5c4f9dd350e54
    ```

    See [the /user endpoint](#the-user-endpoint) below for the response format.
  </Step>
</Steps>

## Token expiry and refresh

Access tokens expire after **7200 seconds (2 hours)**. The `expires_in` field is always included in the token response. End-user-facing clients must interpret this value and follow the refresh process before the token expires.

Signon issues a refresh token alongside every access token. To obtain a new access token without requiring the user to sign in again, exchange the refresh token:

```http
POST /oauth/token HTTP/1.1
Host: signon.publishing.service.gov.uk
Content-Type: application/x-www-form-urlencoded;charset=UTF-8

grant_type=refresh_token
  &refresh_token=8257e65c97202ed1726cf9571600918f3bffb2544b26e00a8a5d5cfe5a04d8
  &client_id=s6BhdRkqt3
  &client_secret=7Fjfp0ZBr1KtDRbnfVdmIw
```

## The /user endpoint

`GET /user` returns the authenticated user's details and permissions. The response is scoped to the application identified by the access token — a client only sees its own permissions, not the full set.

```json
{
  "user": {
    "uid": "fakeuid",
    "name": "Fake User",
    "email": "fake.user@digital.cabinet-office.gov.uk",
    "permissions": ["signin"],
    "organisation_slug": "cabinet-office",
    "organisation_content_id": "d4d0b72a-0d0b-4c6e-87f1-2b0e1f4e9abc",
    "disabled": false
  }
}
```

Each client application only receives its own permissions — the `permissions` array is scoped to the requesting application. If the user is suspended, `permissions` is returned as an empty array and `disabled` is `true`.

<Note>
  The `/user` endpoint validates the access token, records that permissions have been synchronised for the requesting application (`permissions_synced!`), and returns only that application's permissions. An optional `client_id` query parameter can be included; if provided, Signon validates that the token belongs to that application and returns `401` if it does not match.
</Note>

## Permission synchronisation

Signon actively pushes permission changes to client applications. Applications must implement two callback endpoints.

### Permissions changed

When a GDS administrator changes a user's permissions, Signon sends an HTTP `PUT` to `/users/:uid` on every registered application with the updated user object:

```http
PUT /users/fakeuid HTTP/1.1
Host: myapp.com
Content-Type: application/json

{
  "user": {
    "uid": "fakeuid",
    "name": "Fake User",
    "email": "fake.user@digital.cabinet-office.gov.uk",
    "permissions": ["signin"]
  }
}
```

The application **must** update its local record of the user's permissions and change its behaviour accordingly.

### Account suspended

When a user is suspended, Signon sends an HTTP `POST` to `/users/:uid/reauth` on every registered application:

```http
POST /users/fakeuid/reauth HTTP/1.1
Host: myapp.com
```

The application **must** invalidate the user's current session and redirect them to the Signon authorization server.

<Tip>
  The [gds-sso](https://github.com/alphagov/gds-sso) library handles both callbacks consistently across GOV.UK Ruby applications.
</Tip>

## Endpoint reference

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/oauth/authorize` | Authorization endpoint. Redirects user to sign in. |
| `POST` | `/oauth/token` | Token endpoint. Exchanges code or refresh token for access token. |
| `GET` | `/user` | User info endpoint. Returns authenticated user and their permissions. |
| `POST` | `/oauth/access_token` | Legacy alias for `/oauth/token` (OAuth v1 compatibility). |

<Warning>
  The `/oauth/access_token` endpoint exists only for backwards compatibility with older OAuth v1 clients. New integrations should use `/oauth/token`.
</Warning>
